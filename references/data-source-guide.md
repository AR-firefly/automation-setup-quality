# 数据源验证指南

> 无论你使用什么平台、什么工具，设置自动化前的数据源验证遵循相同的原则。
> 本指南提供通用方法论，并附真实平台示例供参考。

---

## 核心原则

1. **绝不假设结构**：每种数据源必须先读取原始结构。你以为的 JSON 结构和实际返回的可能完全不同——"看起来像"不等于"就是"。
2. **逐级验证**：容器节点 → 内容节点 → 字段/记录 → 值格式。跳过任何一级都可能引入错误。
3. **记录三要素**：每次验证记录**类型 + 标识 + 验证结果**。已验证的数据源不需要重复验证。

## 通用验证流程

对任何数据源，动手前先回答这 4 个问题：

1. **这个数据源是什么类型？**
   - 数据库 / API / 文档 / 文件 / 网页？
   - 不同类型需要不同的验证方法

2. **它的原始结构是什么样的？**
   - JSON schema？文档大纲？表格字段列表？
   - **先读原始返回，再写筛选逻辑**——这是最常见的翻车原因

3. **我要的数据在哪个层级？**
   - 某个字段？某个子节点？某个 API 路径？
   - 确认数据存在且格式可解析

4. **筛选和分页处理了吗？**
   - `has_more` 是 true 的话，下一页呢？
   - `obj_type` 跟筛选条件一致吗？不一致就是筛选错了
   - 内容被截断了吗？

## 数据源类型与验证要点

### 文档 / 知识库

| 验证项   | 方法                    | 通过标准                 |
| -------- | ----------------------- | ------------------------ |
| 结构确认 | 读大纲（outline / TOC） | 有实质标题层级，不是空壳 |
| 内容完整 | 读全文（或抽样）        | 内容未截断，链接可解析   |
| 链接跳转 | 追踪引用链接            | 目标存在，类型正确       |

### 数据库 / 多维表格

| 验证项   | 方法               | 通过标准                               |
| -------- | ------------------ | -------------------------------------- |
| 记录总数 | 查询总记录数       | 在预期范围内                           |
| 关键字段 | 检查字段非空率     | 必填字段有值，无大量空值               |
| 字段类型 | 读字段 schema      | 日期字段是日期类型，数字字段是数值类型 |
| 筛选效果 | 带条件查询 vs 全量 | 筛选后数量符合预期                     |

### API

| 验证项    | 方法                          | 通过标准               |
| --------- | ----------------------------- | ---------------------- |
| 返回结构  | 发一次请求读原始 JSON         | 结构如预期，无 403/500 |
| 分页机制  | 检查 `has_more` / `next_page` | 知道如何处理下一页     |
| 筛选条件  | 带参数请求 vs 无参数          | 参数生效，结果匹配预期 |
| 限流/频率 | 确认 API 调用限制             | 在自动化运行频率范围内 |

### 网页

| 验证项   | 方法                               | 通过标准             |
| -------- | ---------------------------------- | -------------------- |
| 可访问性 | 请求页面                           | 返回 200，非 403/500 |
| 内容类型 | 检查页面结构                       | 静态 HTML vs JS 渲染 |
| JS 渲染  | 如需渲染，确认工具有 headless 能力 | 内容可完整获取       |

### 本地文件

| 验证项 | 方法         | 通过标准                       |
| ------ | ------------ | ------------------------------ |
| 存在性 | 确认路径可读 | 文件存在                       |
| 编码   | 检查文件编码 | UTF-8 优先，其他编码需特殊处理 |
| 大小   | 确认文件大小 | 在预期范围内，非空文件         |
| 格式   | 尝试解析     | 格式正确（JSON/CSV/TXT 等）    |

## 示例：飞书 / Lark（Feishu）

以下以飞书为例，展示通用原则如何落地到具体平台。

### wiki 节点

```
# 获取节点信息（确认类型）
→ 检查 obj_type：wiki / bitable / docx
→ 检查 has_child：是否有子节点
→ 检查 has_more：是否还有下一页

# 列出子节点（如需遍历）
→ 逐页读取，直到 has_more = false
```

**确认要点**：`obj_type` 是不是你要的类型？`has_more` 处理了吗？`parent_node_token` 是空的说明是根节点。

### 多维表格（bitable）

```
# 列出所有数据表
→ 确认 app_token 下有哪几张表

# 列出字段
→ 确认字段名、类型、是否必填

# 搜索记录
→ 按筛选条件搜索
→ 确认总记录数、关键字段非空
```

**常见错误**：把 bitable 当文件夹扫——bitable 是一张表，不是文件夹。先看 `obj_type` 再决定怎么处理。

### 文档（docx）

```
# 读大纲
→ 确认是否有实质标题结构

# 读全文
→ 确认内容是否截断
```

**注意**：如果通过 wiki URL 访问文档，需要先解析 URL 取出 docx token，再读内容。

> 提示：飞书 API 会截断内容片段。如果概要看起来很短，检查是不是 truncated。

## 附录：创建你自己的验证表

读数据源前，先填这张空白表。填完再动手写任何自动化逻辑。

| 数据源标识 | 类型 | 原始结构命令 | 预期值 | 实际值 | 通过? |
| ---------- | ---- | ------------ | ------ | ------ | ----- |
|            |      |              |        |        |       |
|            |      |              |        |        |       |

### 使用方法

1. **数据源标识**：填入数据源唯一标识（文件路径、API URL、文档 token 等）
2. **类型**：文档 / 表格 / API / 文件 / 网页
3. **原始结构命令**：用什么命令/工具读取原始结构（如 Read、WebFetch、API 请求）
4. **预期值**：你假设应该看到什么
5. **实际值**：实际返回了什么
6. **通过?**：预期 vs 实际一致 → 通过 ✅。不一致 → 你的假设错了，先修正

# 自动化设置质量保证

写自动化之前，先过这三步。缺一步不准开始。

---

## 为什么搞这套东西

大多自动化翻车不是因为代码写错，是**假设错了**：

- 把多维表格当文件夹遍历
- 假设 API 返回 `{ items: [...] }`，实际是 `{ data: { repositories: [...] } }`
- 信了预设的 skip 标记，但那个标记已经过时了
- 没看 `has_more`，只拿了一页数据

这些全可以在写第一行代码之前就发现。这个流程就是干这个的。

---

## 流程

### Step 0：先看能不能复用

| 情况                    | 怎么跳                       |
| ----------------------- | ---------------------------- |
| 同一批数据源之前验证过  | 跳过 Step 1，从 Step 2 开始  |
| "老样子""和上次一样"    | 跳过 Step 1+2，直接 Step 3   |
| "你先动""直接干"        | 跳过 Step 2，Step 1 完直接写 |
| 第一次搞 / 数据源不一样 | 完整走一遍                   |

验证过的数据源打个标记：`<类型>:<标识>`。下次碰到同样的就直接复用。

### Step 1：验证数据源

每条数据源先读原始结构。**不准假设你还没确认的东西。**

验证什么：

- 原始结构长什么样（JSON schema / 大纲 / 字段列表）
- 类型判断对不对（文档 vs 表格 vs API vs 文件夹）
- 筛选条件生没生效（带条件查一次 vs 全量查一次）
- 有下一页吗？内容截断了吗？

搞完对照检查：

- 数据拿全了？`has_more` 是 true 的话下一页呢？
- 节点类型跟筛选条件一致吗？不一致重来。
- 字段名和预期一样吗？

> 具体怎么验各种数据源 → `references/data-source-guide.md`

### Step 2：确认方案

方案说清楚四件事：

1. **数据来源**：token / URL / 路径
2. **筛选条件**：范围 + 时间 + 过滤规则
3. **评估标准**：什么算过，什么算不过
4. **输出格式**：推消息 / 写报告 / 打标记 / 不说话

需求方说"不对"→ 退回 Step 1。
说"你先动"→ 直接写。
没回音 → **再问一次**，不准自动通过。

### Step 3：验证交付

写完了逐项验，过了再交：

| 验什么     | 怎么验                 | 过线标准                 | 能自动吗         |
| ---------- | ---------------------- | ------------------------ | ---------------- |
| 数据源可读 | 再跑一次读取命令       | 返回正常结构，没 403/500 | ✅ 直接跑        |
| 筛选对了   | 对比预期条数和实际条数 | 数量/范围对得上          | ✅ 数一下        |
| 逻辑可行   | 读一条内容人工判断     | 能基于标题+大纲做判断    | ❌ 得人看        |
| 格式对了   | 检查输出结构           | 符合要求                 | ✅ 数段落/关键词 |

不过就退回 Step 1。最多 3 轮。

3 轮还过不了 → **不交**，告诉需求方"3 轮验证没通过，原因是..."，让他们定下一步。

## 反面案例

真实翻车记录。全是"我以为……实际上……"。

### 把表格当文件夹扫

没看 `obj_type`，把 406 条记录的表当 3 个节点的文件夹扫了。遍历跑完才发现不对。

**教训**：先看节点类型再决定怎么遍历。不是所有 URL 背后都是同一种东西。

### 信了预设标记

数据源文件里有 skip 标记，自动化直接跳了没验。结果那个标记早过期了，该处理的全被跳了。

**教训**：skip 必须带理由。没理由的 skip 等于没处理。预设标记也要验。

### API 结构猜错了

扫 GitHub repo，假设 API 返回 `{ items: [...] }`，实际是 `{ data: { repositories: [...] } }`。筛选条件按猜的结构写的，一条都没匹配上，白跑一轮。

**教训**：先发个请求看看原始返回长什么样再写筛选。"看起来像 GitHub API 就会返回 items"——这种假设不值钱。

## 写作规范

### DO

- 给具体的东西：token、命令、完整链路
- 每种输出类型写清楚怎么处理
- if A then B——那 if not A 呢？
- tracking 写在最后，别忘了

### DON'T

- 不验证就假设结构
- 默认读者懂缩写
- 逻辑不闭环（条件成立怎么办？不成立怎么办？）

### rrule 模板

```
# <名称>

## 任务目标
一句话说什么时间做什么事。

## 调度规则
- FREQ=DAILY/WEEKLY/MONTHLY[;BYDAY=MO,TU,WE,TH,FR]
- 时间：北京时间 HH:MM（UTC BYHOUR=XX;BYMINUTE=XX）
- 工作目录：<项目根目录路径>

## 操作步骤

### Step 1：读上下文
- 读 tracking 文件 / 记忆文件
- 读配置文件

### Step 2：取数据
- 具体命令 / 工具调用
- 筛选条件

### Step 3：处理
- 判断逻辑
- 评估标准

### Step 4：输出
- 有结果怎么推
- 没结果怎么静默

## 关键约束
- 特殊身份要求（如 --as user）
- 去重要求
- 禁止做的事
```

---

## 约束

- 三步缺一步不准动手
- 验证最多 3 轮，过不了就报错
- 喊停立刻停
- 能自动验的用命令跑，不准说"看起来没问题"
